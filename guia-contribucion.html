Sección 1

Ramas

Nombramos las ramas con prefijo + descripción corta en kebab-case. Una rama = un cambio atómico.

PrefijoCuándo usarloEjemplo
feat/ Nueva funcionalidad feat/demo-modal-per-user
fix/ Corrección de bug fix/nav-link-playbooks
chore/ Mantenimiento, deps, limpieza chore/cleanup-legacy-html
docs/ Documentación docs/readme-deploy
design/ Cambios visuales o de branding design/liquid-glass-v3

Reglas:

  • Las ramas salen de master y vuelven a master.
  • Si tu rama hace dos cosas no relacionadas, la partes en dos.
  • Borramos la rama después del merge.
Sección 2

Commits

Usamos Conventional Commits en español. Formato:

<tipo>(<alcance>): <descripción en imperativo>

Tipos: feat fix chore docs design refactor perf

Ejemplos reales del proyecto:

feat(team): agrega a Giancarlo Vilcapoma como CTO
fix(nav): corrige slug roto en link de Playbooks
design(hero): migra hero section a Liquid Glass v3
chore(repo): elimina archivos HTML legacy del root

Reglas:

  • Un commit = un cambio lógico. Si escribiste "y también", son dos commits.
  • Imperativo presente. "agrega", no "agregué" ni "agregado".
  • No commits WIP a master. Usa git stash o rama draft para trabajo parcial.
Sección 3

Pull Requests

Antes de abrir el PR

  • Hiciste rebase sobre master (no merge de master en tu rama)
  • Probaste el cambio manualmente — no asumas que "debería funcionar"
  • Si es HTML/CSS/JS, lo abriste en el navegador y verificaste visualmente
  • Si es infraestructura (DNS, Caddy, Docker), lo probaste en el VPS o documentaste por qué no
  • Hiciste squash de commits "wip", "typo", "test" en commits con sentido
  • El diff es limpio — sin archivos de prueba, backups, ni ruido

Estructura de la descripción

Todo PR responde qué, por qué y cómo. No asumas que el revisor tiene contexto.

## Qué
Una o dos frases que describan el cambio.

## Por qué
El problema que resuelve, el issue que cierra, o la decisión de producto.
Incluye screenshots del antes/después si es un cambio visual.

## Cómo
Decisiones técnicas: por qué esta solución, trade-offs, archivos clave.

## Cómo probar
1. Ir a /ruta
2. Hacer clic en X
3. Verificar que Y

Sección 4

Code Review

Para el autor

  • El PR es una conversación, no una entrega. El revisor es tu par. Su trabajo es encontrar problemas antes de producción.
  • No te pongas a la defensiva. Si el revisor no entendió algo, el código no es suficientemente claro. Arregla el código, no discutas.
  • Responde a todos los comentarios. Un "listo" o "hecho" basta.
  • No hagas force-push después del review. Agrega commits nuevos. El squash se hace al final.

Para el revisor

  • Revisa el diff, no la persona. "Este loop es O(n²)" — no "escribiste mal este loop".
  • Distingue bloqueante de sugerencia. Usa nit: o sugerencia: para preferencias personales.
  • No dejes pasar cosas rotas por ser amable. El código que mergeamos hoy es el incidente de mañana.
  • Aprueba cuando esté listo. Nada de "LGTM pero…" — si hay peros, son comentarios, no aprobación.

Sección 6

Anti-patrones

Anti-patrón Por qué es dañino Qué hacer en vez
"Refactor mientras agrego feature" Diff gigante, imposible de revisar Rama separada de refactor, mergear primero
Commits "wip", "test", "asdf" Historia del repo ilegible Squash antes de pushear a remoto
PRs de 20+ archivos sin descripción Nadie los revisa bien Partir en PRs más chicos o documentar excepcionalmente
"Funciona en mi máquina" El VPS no es tu máquina Probar en entorno similar a prod
Mergear sin review porque "es chiquito" Bugs más caros vienen de cambios de 1 línea Todo PR lleva review, sin excepciones
Deploy a prod un viernes a las 6pm Nadie quiere debuggear un sábado Lunes a jueves, antes de las 4pm
Commits en inglés y español mezclados Log inconsistente e ilegible Todo en español, Conventional Commits
Sección 7

Herramientas & Referencia rápida

Comandos esenciales

git checkout -b feat/mi-cambio          # crear rama
git add -p                              # stage interactivo (recomendado)
git commit -m "feat(scope): descripción" # commit con formato
git rebase master                       # actualizar contra master
git push origin feat/mi-cambio          # subir rama
# Abrir PR en GitHub → pegar descripción generada por AI
# Esperar review → responder comentarios → merge → borrar rama

Toolkit

  • Git: git push --force-with-lease — nunca --force solo.
  • AI: Usa los prompts de la Sección 5 con Claude, ChatGPT o Copilot.
  • Screenshots: Cmd+Shift+4 en macOS, pegar directo en GitHub.
  • Pre-commit: Revisión manual por ahora. Si el proyecto crece, agregaremos hooks.
